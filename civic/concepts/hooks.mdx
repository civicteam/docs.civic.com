Hooks provide a middleware layer for intercepting and processing MCP requests and responses. They enable you to add custom logic, security rules, or transformations without modifying the underlying MCP servers.

## How Hooks Work

```
AI Assistant → [Request Hook] → MCP Server
                                     ↓
AI Assistant ← [Response Hook] ← MCP Server
```

Each hook in the chain can:

- **Inspect** the request or response data
- **Modify** parameters or results
- **Block** requests based on custom rules
- **Enrich** data with additional context

## Common Hook Patterns

| Pattern | Description |
|---|---|
| **Security Filtering** | Validate requests against security policies before forwarding |
| **Rate Limiting** | Control request frequency and volume |
| **Data Transformation** | Modify request or response formats |
| **Audit Logging** | Record all tool interactions for compliance |
| **Content Filtering** | Remove or redact sensitive information from responses |
| **Tool Filtering** | Block specific tools from appearing in the agent's tool list |

## Hooks and Guardrails

Civic's [Guardrails](/civic/concepts/guardrails) system is built on the hook architecture. When you configure guardrails — blocking specific tools, presetting parameters, or filtering responses — these are implemented as hooks applied to the request pipeline.

For most use cases, the Guardrails UI and Configurator Agent provide sufficient control without needing to write custom hooks directly.
